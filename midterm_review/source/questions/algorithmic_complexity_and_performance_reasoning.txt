slug: identify-inefficient-list-membership-in-genotype-check
difficulty: medium
topic: Comparing implementations of the same task to identify algorithmic inefficiency
question: |
  In a strain-genotype analysis pipeline, one step stores a strain genotype as a
  collection of allele names, and another step checks that genotype against a
  panel of allele names to flag.

  Assume the following variables are already defined:

  <python>
  strain_alleles = ['recA1', 'endA1', 'gyrA96', ...]              # Python list of allele names (length n)
  flagged_alleles = {'gyrA96', 'rpsL150', 'lacIq', ...}           # Python set of allele names (length m)
  </python>

  The goal is to print each allele name that appears in both collections.

  Assume <python>strain_alleles</python> contains no duplicates, output order does
  not matter, and ignore the cost of printing itself.

  Which option is the most algorithmically inefficient for many membership
  checks?
choices:
  A: |
    Convert the strain-allele list to a set first, then test flagged alleles against that set.

    <python>
    strain_allele_set = set(strain_alleles)
    for allele in flagged_alleles:
        if allele in strain_allele_set:
            print("flagged allele present:", allele)
    </python>
  B: |
    Test each flagged allele directly against the strain-allele list.

    <python>
    for allele in flagged_alleles:
        if allele in strain_alleles:
            print("flagged allele present:", allele)
    </python>
  C: |
    Convert the strain-allele list to a set, collect matches with a comprehension, then print.

    <python>
    strain_allele_set = set(strain_alleles)
    hits = [allele for allele in flagged_alleles if allele in strain_allele_set]
    for allele in hits:
        print("flagged allele present:", allele)
    </python>
  D: |
    Scan the strain-allele list once and test membership in the flagged-allele set.

    <python>
    for allele in strain_alleles:
        if allele in flagged_alleles:
            print("flagged allele present:", allele)
    </python>
answer: B
explanation: |
  Option B is the most algorithmically inefficient because it repeatedly checks
  whether an element is in a Python list (<python>allele in strain_alleles</python>)
  inside a loop. If <python>len(strain_alleles) = n</python> and
  <python>len(flagged_alleles) = m</python>, each list membership test can take
  O(n) time in the worst case, so repeating it m times gives O(mn) total runtime.

  The main punchline is that repeated membership tests should generally not be
  done against a list when a set is available or can be built once.

  - A builds a set once, then uses typically O(1) average-case set membership.
  - C has the same asymptotic idea as A, just written in a different style.
  - D scans the list once and tests membership in <python>flagged_alleles</python>,
    which is already a set, so it avoids repeated list membership checks.

  The inefficiency targeted by this question is repeated
  <python>x in some_list</python> tests.

  A function scans a DNA sequence once to count how many G and C bases it
  contains:

  <python>
  def gc_count(seq):
      count = 0
      for base in seq:
          if base in 'GC':
              count += 1
      return count
  </python>

  If <python>n = len(seq)</python>, what is the Big-O runtime of this function?
choices:
  A: O(1) - the number of operations is constant regardless of sequence length
  B: O(log n) - the loop grows logarithmically with sequence length
  C: O(n) - the function examines each base once in a single pass
  D: O(n^2) - the if statement makes the loop quadratic in sequence length
answer: C
explanation: |
  The loop visits each base in <python>seq</python> once, so the total work
  grows linearly with sequence length. The membership test
  <python>base in 'GC'</python> is over a constant-size string, so it does not
  change the asymptotic runtime. Therefore the runtime is O(n).

question: |
  A function compares every sequence in a list to every other sequence to find
  exact duplicates:

  <python>
  seqs = [...]  # length n
  duplicates = []
  for i in range(len(seqs)):
      for j in range(i + 1, len(seqs)):
          if seqs[i] == seqs[j]:
              duplicates.append((i, j))
  </python>

  Ignoring the cost of string comparison itself and focusing on the loop
  structure, what is the Big-O runtime in terms of <python>n</python>?
choices:
  A: O(1) - the inner loop starts at i+1, so the total work is bounded
  B: O(n) - each sequence is touched once by the outer loop only
  C: O(n log n) - nested loops imply logarithmic growth in the inner loop
  D: O(n^2) - the code performs a pairwise comparison pattern over the list
answer: D
explanation: |
  The code compares many pairs of items using nested loops. The outer loop runs
  about n times, and the inner loop runs a decreasing but still linear number of
  times overall, giving a total proportional to n(n-1)/2. That scales as O(n^2).

  </python>

  B: O(n log n) - mixed polynomial terms imply a logarithmic factor
  C: O(n^2) - the highest-order term determines asymptotic growth
  D: O(3n^2 + 20n + 7) - Big-O should keep the exact full expression
