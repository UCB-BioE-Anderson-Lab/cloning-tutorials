slug: variable-scope-function-local
difficulty: easy
topic: Understanding local vs. global variable scope and why inner assignments do not affect outer names
question: |
  A lab utility module defines a default promoter at the top level, then
  calls a helper that appears to update it:

  <python>
  promoter = 'J23119'

  def swap_promoter():
      promoter = 'J23101'

  swap_promoter()
  print(promoter)
  </python>

  What does <python>print(promoter)</python> output?
choices:
  A: "'J23101'"
  B: "'J23119'"
  C: "None"
  D: "UnboundLocalError"
answer: B
explanation: |
  Assigning to a bare name inside a function creates a new local variable
  that shadows the outer name for the duration of that call. The module-level
  <python>promoter</python> is never touched. After <python>swap_promoter()</python>
  returns, <python>promoter</python> still refers to <pre>J23119</pre>.
  To modify the outer name, the function would need a <python>global promoter</python>
  declaration.

slug: string-method-returns-new-object
difficulty: easy
topic: Understanding that string methods return new objects and do not modify the original
question: |
  A reverse complement utility uppercases an input sequence before processing:

  <python>
  seq = 'atgccc'
  seq.upper()
  print(seq)
  </python>

  What does <python>print(seq)</python> output?
choices:
  A: "'ATGCCC'"
  B: "'atgccc'"
  C: "None"
  D: "TypeError"
answer: B
explanation: |
  String methods do not modify the string they are called on — strings are
  immutable. <python>seq.upper()</python> returns a new string, but that
  return value is discarded here. <python>seq</python> still refers to the
  original lowercase string. To capture the result, the caller would need
  <python>seq = seq.upper()</python>.
