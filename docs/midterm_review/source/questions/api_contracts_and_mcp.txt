slug: calc-molarity-hidden-state-bug
difficulty: medium
topic: Recognizing how mutating shared state during run breaks repeatability and violates a clean computation contract
question: |
  What is wrong with this code?

  <python>
  class CalcMolarity:
      def __init__(self):
          self.correction_factor = 1.0

      def run(self, moles: float, liters: float, temperature: float) -> float:
          if temperature > 25:
              self.correction_factor += 0.01 * (temperature - 25)
          return (moles / liters) * self.correction_factor
  </python>

choices:
  A: The method should return the correction factor too, so callers can verify the temperature adjustment each time
  B: The temperature logic should be in __init__, so setup and validation happen before any calculation runs
  C: The method should avoid floats here, because repeated decimal math makes scientific results drift over time
  D: The method mutates shared state while computing, so later outputs depend on prior calls, not just inputs
answer: D
explanation: |
  The flaw is that <python>run()</python> changes <python>self.correction_factor</python>
  while computing the result. Because that state persists on the object, later
  calls can produce different outputs for the same inputs depending on what was
  run earlier.

  For example, a call with temperature 30 increases the correction factor. A
  later call at 20C applies no new correction, but still uses the previously
  modified factor. Another call at 30C adds correction again on top of the
  already changed value. This breaks repeatability and makes the method behave
  like a stateful accumulator rather than a clean computation from explicit
  inputs.

  The core issue is hidden state mutation during <python>run()</python>, not the
  use of classes, methods, or floating-point numbers.
